# Add thread acquire+release for ZeroMQ msg calls
author: vtnerd
created_at: 2025-09-22T23:57:26Z
status: open

I ran thread-sanitizer on `monero-lws` unit tests and the initial results were horrifying.

After digging into the report, I determined the primary cause to be internal to ZeroMQ. The library has a lock-free queue that uses atomics, but no explicit memory fence for the data is provided. Theoretically, on probably only alpha cpus via dependent load reordering, this could cause memory issues.

Additionally, the C++ memory fencing utilities are not supported by TSAN! So, we have to manually call into the library with a memory address for tracking.

NOTE: Yes, ZeroMQ on my system was **not** compiled with TSAN magic, however, I believe there to be an issue after looking at the source.
