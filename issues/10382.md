# p2p: fix hanging shutdown
author: j-berman
created_at: 2026-03-25T22:40:51Z
status: closed

@nahuhh reported that the latest `monerod` (v0.18.4.6) can hang on shutdown sporadically (basicswap's CI tests started failing). @nahuhh also identified a way to reproduce by starting with `monerod --regtest`, letting it run for 10 seconds, then ctrl+c.

This hang-on-exit was introduced in #10278.

### The problem

ctrl+c sends a stop signal to the daemon, which stops the `io_context` [here](https://github.com/monero-project/monero/blob/b9998fc9e1c280d30e4b40ba983b1964eb256d66/contrib/epee/include/net/abstract_tcp_server2.inl#L1558).

If the server stops the `io_context` while it's in the middle of invoking a handshake to another connection and waiting for a response, then [this callback](https://github.com/monero-project/monero/blob/b9998fc9e1c280d30e4b40ba983b1964eb256d66/src/p2p/net_node.inl#L1181-L1237) never executes, blocking [here](https://github.com/monero-project/monero/blob/b9998fc9e1c280d30e4b40ba983b1964eb256d66/src/p2p/net_node.inl#L1241) and causing the daemon to hang on exit (the thread handling the handshake invocation blocks).

Prior to #10278, [this cancel_handler](https://github.com/monero-project/monero/blob/b9998fc9e1c280d30e4b40ba983b1964eb256d66/contrib/epee/include/net/abstract_tcp_server2.inl#L755) would execute synchronously when the server processed the stop signal [here](https://github.com/monero-project/monero/blob/b9998fc9e1c280d30e4b40ba983b1964eb256d66/src/p2p/net_node.inl#L1160), immediately firing the callback in the handshake invocation mentioned above [here](https://github.com/monero-project/monero/blob/b9998fc9e1c280d30e4b40ba983b1964eb256d66/contrib/epee/include/net/levin_protocol_handler_async.h#L241). But because #10278 now ensures an async shutdown sequence that executes inside the strand, it requires the `io_context` to be running to execute to completion when `close()` is called.

The unit test aims to mirror what's happening when `monerod` hangs (without modifying the server's internal logic). If you call `close(<connection id>, false)` in the unit test, don't pass `close_all_connections` into `send_stop_signal`, and call `close_all_connections` in the line after `send_stop_signal`, then the test hangs (which is essentially what's happening when `monerod` hangs).

### The solution

When shutting the server down, make sure to **wait** for all connections to complete the shutdown sequence **before** stopping the `io_context`.

This guarantees that the `io_context` stays running until a connection's shutdown sequence executes to completion inside the strand. We still want the shutdown sequence to execute inside the strand because of the issue pointed out in #10278.

### Future considerations

It's possible that some connection gets queued to the strand for shutdown, then erased from the connections map before its shutdown sequence completes [here](https://github.com/monero-project/monero/blob/b9998fc9e1c280d30e4b40ba983b1964eb256d66/contrib/epee/include/net/levin_protocol_handler_async.h#L672-L709). If the `io_context` then stops during server shutdown before that shutdown sequence executes, then I figure it may still be possible the shutdown sequence may not complete for that connection. So long as there is no other blocking call that the server is waiting on for that connection, then this may not be an issue.

If this surfaces as an issue (continued sporadic hangs on exit caused by the above explanation), then a way to solve this could be to **always** wait for a connection's shutdown sequence to complete when deleting the ref to a connection, thus we can guarantee that all connections are properly shut down before we stop the `io_context`.  This would be a more involved change.

Another way to solve this may be to have `cancel_handler` run outside the strand, but that presents a whole other set of more complicated changes.

The goal with this PR as is is to solve this currently known hang-on-exit issue in a fairly isolated way (it should only affect server shutdown).
