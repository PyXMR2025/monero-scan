# Add json output support to ::wire and partially use in ZMQ-PUB
author: vtnerd
created_at: 2025-11-29T21:28:52Z
status: open

The community has accepted my write/output proposal for objects, but not the read proposal (which is _way_ more complicated). This adds a json output using this new system, and uses it in _some_ of the ZMQ-PUB subsystems.

The next step is adding msgpack (or cbor), such that ZMQ-REP uses binary for some of the responses, and ZMQ-PUB uses msgpack/cbor for some of the notifications. The biggest benefit will be a reduction in scan time in LWS by at least 11%, with more gains possible as features are added. The second benefit will be slightly lower latency in mining notifications.

### Background

I've done an experimental build where `monerod` returns blocks via msgpack where each tx+block is encoded as one giant blob (as opposed to individual fields). The performance improvement for LWS scanning is ~11%. This is faster than HTTP-binary because ZMQ is a faster protocol as well (everything is binary). The 11% is conservative, as there are more cycle reduction in using integer keys and binary requests (my experimental branch had requests in json but responses in msgpack).

### Edit
The 11% improvement is for **total scan time**, not just the unpack performance. So this isn't just a small niche thing for LWS, sadly.

Integer keys speed up scan performance by ~0.97%, so it's not clear the feature is worth the extra complexity.
